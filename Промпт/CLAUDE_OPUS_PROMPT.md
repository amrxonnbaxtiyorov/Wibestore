## СТЕК ТЕХНОЛОГИЙ — КОНТЕКСТ ДЛЯ ПОНИМАНИЯ

**Backend (Django / Python):**
- Django REST Framework (DRF)
- JWT-аутентификация через `djangorestframework-simplejwt`
- `UserProfileUpdateSerializer` — PATCH `/api/v1/profile/` для обновления профиля

- `useAuth()` — контекст с данными пользователя (`user`, `isAuthenticated`)
- `AvatarEditModal.jsx` — модальное окно выбора фото профиля
- `ProfilePage.jsx` — страница профиля, вызывает `AvatarEditModal`

## БАГ №1 — КРИТИЧЕСКИЙ: НЕВОЗМОЖНОСТЬ СМЕНЫ ФОТО ПРОФИЛЯ

### Точное описание симптома
После регистрации пользователь переходит в профиль `/profile`, нажимает кнопку редактирования аватара (иконка карандаша), открывается модальное окно `AvatarEditModal`, пользователь выбирает фото и нажимает "Tasdiqlash" (Подтвердить). Фото остаётся неизменным. Никакой ошибки не показывается. Никакого запроса к API не происходит.

### Диагностика — точные причины бага

**Причина 1 — `AvatarEditModal` не подключён к API вообще.**

Файл `src/components/AvatarEditModal.jsx`:

```jsx
const handleSave = () => {
    if (cropped && onSave) onSave(cropped); // <-- передаёт base64 строку
    onClose();
};
```

Компонент вызывает `onSave(cropped)`, передавая base64 строку изображения. Но никакого API-вызова нет ни внутри `AvatarEditModal`, ни снаружи.

**Причина 2 — `ProfilePage` вызывает `AvatarEditModal` без пропа `onSave`.**

Файл `src/pages/ProfilePage.jsx`:

```jsx
{showAvatarModal && (
    <AvatarEditModal onClose={() => setShowAvatarModal(false)} />
)}
```

Проп `onSave` не передаётся. Даже если он был бы реализован правильно — callback всё равно не придёт.

**Причина 3 — Отсутствует хук для загрузки аватара.**

Функция `useUpdateProfile()` из `useProfile.js` делает `PATCH /profile/` без `Content-Type: multipart/form-data`, передавая JSON. Но `avatar` — это `ImageField`, который требует `multipart/form-data`. Передача base64 строки в JSON-теле не сохранит файл в модели.

Backend `UserProfileUpdateSerializer` включает поле `avatar` (строка 153 в `serializers.py`), но принимает его только как файл.

### Требуемое решение — полный и точный план

**Шаг 1: Переписать `AvatarEditModal.jsx`.**

Компонент должен:
- Принимать проп `onClose` (без `onSave` — логика загрузки должна быть внутри)
- Импортировать `useUploadImage` и `useUpdateProfile` из хуков
- При выборе файла — хранить не только base64 превью, но и сам `File` объект
- При нажатии "Подтвердить":
  1. Создать `FormData`, добавить файл: `formData.append('image', file)`
  2. Вызвать `uploadImage(formData)` через `useUploadImage`
  3. После успешной загрузки получить `data.url`
  4. Вызвать `updateProfile({ avatar_url: data.url })` через `useUpdateProfile` (или передать файл напрямую через multipart в PATCH `/profile/`)
  5. Инвалидировать `AuthContext` — обновить данные пользователя
  6. Показать toast "Rasm muvaffaqiyatli o'zgartirildi"
  7. Закрыть модальное окно
- Показывать состояние загрузки (`isPending`) на кнопке
- Обрабатывать ошибки и показывать toast с текстом ошибки

**Важный нюанс по backend:**
- `ImageUploadView` (`POST /upload/image/`) возвращает `{ url, thumbnail_url }` — URL внешней загрузки
- После получения URL нужно обновить `avatar_url` пользователя через `PATCH /profile/ { avatar_url: "..." }`
- `UserProfileUpdateSerializer` содержит поле `avatar`, но не `avatar_url` — нужно проверить и **добавить `avatar_url` в поля сериализатора**, если его там нет
- `AuthContext` должен перезапросить данные пользователя после успешного обновления — нужно вызвать `queryClient.invalidateQueries(['profile'])` и обновить данные в контексте

**Шаг 2: Обновить `ProfilePage.jsx`.**

```jsx
{showAvatarModal && (
    <AvatarEditModal onClose={() => setShowAvatarModal(false)} />
)}
```

`AvatarEditModal` теперь управляет всей логикой самостоятельно. Пропа `onSave` не нужно. Нужно убедиться, что после закрытия модалки — аватар в профиле обновляется (через реактивные данные из `useAuth()` или `useProfile()`).

**Шаг 3: Убедиться, что `UserProfileUpdateSerializer` принимает `avatar_url`.**

Файл `wibestore_backend/apps/accounts/serializers.py`, класс `UserProfileUpdateSerializer` — добавить `avatar_url` в список `fields`:

```python
fields = [
    "username",
    "full_name",
    "phone_number",
    "avatar",
    "avatar_url",  # <-- добавить
    "language",
    "timezone",
]
```

**Шаг 4: Убедиться, что `ProfileView` принимает `multipart/form-data`.**

`ProfileView` наследует от `generics.RetrieveUpdateAPIView`. DRF по умолчанию умеет обрабатывать `multipart`. Убедиться, что `parser_classes` не ограничивает только JSON.

## БАГ №2 — КРИТИЧЕСКИЙ: ОШИБКА ПРИ ПУБЛИКАЦИИ ОБЪЯВЛЕНИЯ

### Точное описание симптома
Пользователь регистрируется, переходит в `/sell`, проходит квиз правил (`SellerRulesQuiz`), заполняет все три шага формы и нажимает кнопку "Публикация". Сверху появляется toast с сообщением об ошибке.

### Диагностика — возможные причины (нужно проверить и исключить каждую)

**Гипотеза A — JWT токен не передаётся при первом запросе после регистрации.**

После регистрации `AuthContext` сохраняет токен. `apiClient` должен добавлять его в заголовок `Authorization: Bearer <token>`. Нужно проверить:
- Как `apiClient` получает токен — из `localStorage`, из контекста, из interceptor'а
- Есть ли race condition: пользователь зарегистрировался, токен ещё не попал в `apiClient`, а запрос уже уходит
- Проверить файл `src/lib/apiClient.js` на наличие interceptor'а и то, откуда он берёт токен

**Гипотеза B — Игры не загружены из API, выбрана mock-игра со строковым ID вместо UUID.**

В `SellPage.jsx` есть явная защита (строки 138–146):

```javascript
const uuidRegex = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
if (formData.gameId && !uuidRegex.test(formData.gameId)) {
    addToast({
        type: 'error',
        message: "O'yinlar ro'yxati yuklanmadi. Sahifani yangilang.",
    });
    return;
}
```

Это **подтверждённая причина**: если `useGames()` не успел загрузить игры из API при первом открытии страницы, используются mock-данные с нестандартными ID (не UUID). Пользователь выбирает игру из mock-списка, ID не проходит UUID-валидацию — показывается ошибка.

**Гипотеза C — Backend возвращает ошибку валидации.**

Если UUID корректный, но backend отвечает 400 или 403 — нужно понять конкретное сообщение. `SellPage.jsx` умеет парсить форматы: `{ error: { code, message, details } }`, `{ detail }`, строку. Нужно убедиться, что ошибка корректно отображается.

**Гипотеза D — Отсутствует email верификация, и backend отклоняет запрос.**

Backend может отклонять создание листинга от неверифицированного пользователя. Нужно проверить, есть ли подобный check в `ListingListCreateView`.

### Требуемое решение — полный и точный план

**Шаг 1: Диагностика загрузки игр.**

Проверить `useGames()` хук — как он загружает данные, есть ли состояние `isLoading`. В `SellPage.jsx` добавить визуальную блокировку: пока игры не загружены (`!gamesData || isLoading`) — не показывать форму или показывать skeleton + сообщение "O'yinlar yuklanmoqda...". Не давать пользователю выбирать игру из mock-списка, если API данные ещё не пришли.

**Шаг 2: Исправить логику fallback на mock-игры.**

Если API вернул игры — использовать только API-игры. Если API не вернул (ошибка, недоступность) — показать сообщение об этом, а не молча подставить mock-игры с невалидными ID.

**Шаг 3: Проверить и исправить `apiClient.js`.**

Убедиться, что:
- Токен читается из `localStorage` через `interceptor` при каждом запросе (не один раз при инициализации)
- Если токен обновляется в `AuthContext`, он должен также обновляться в `localStorage`
- Нет ситуации, где `apiClient` работает со старым или пустым токеном

**Шаг 4: Добавить подробное логирование ошибок.**

Временно (для диагностики) добавить в `onError` листинга `console.error('Listing creation error:', error.response?.data)`. Это позволит точно увидеть, что именно возвращает backend.

## ОБЯЗАТЕЛЬНЫЕ ТРЕБОВАНИЯ К РЕАЛИЗАЦИИ

### Качество кода

1. **Не трогать чужой код без необходимости.** Изменения должны быть минимальными и хирургическими. Если проблема в 3 строках — меняй 3 строки, а не переписывай файл целиком.

2. **Сохранять существующий стиль кода.** Проект использует inline-стили, hook-based архитектуру, React Query для всех API-вызовов. Не нарушать эти паттерны.

3. **Обрабатывать все состояния.** Каждый API-вызов должен обрабатывать: `isPending` (загрузка), `onSuccess` (успех + toast), `onError` (ошибка + toast с понятным сообщением).

4. **Не создавать дублирующих хуков.** Использовать уже существующие `useUploadImage`, `useUpdateProfile`, `useCreateListing`. Если нужен новый хук — добавить его с соблюдением существующих паттернов в `src/hooks/`.

5. **Обновлять AuthContext после изменения аватара.** Пользователь должен видеть новый аватар сразу, без перезагрузки страницы. Это требует инвалидации кеша или принудительного обновления данных пользователя.

### Backend изменения

6. **Если требуется изменение в сериализаторе** — убедиться, что поле добавлено корректно, с правильным типом (`URLField`, `required=False`).

7. **Не ломать существующие API-контракты.** Если добавляешь поле в сериализатор — убедись, что оно не обязательное (`required=False`), чтобы не сломать существующих клиентов.

8. **Проверить CORS.** Если frontend и backend на разных доменах — убедиться, что `multipart/form-data` запросы проходят через CORS без блокировки.
